# Wonderland i18n + Translation Architecture Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Make every site-visible content item available in both Chinese and English, with OpenAI-assisted generation of the missing counterpart when the author writes in either language.

**Architecture:** Store both language versions as first-class Typst content files, but enforce pairing through a stable language-neutral `i18nKey` and deterministic checks. OpenAI translation runs only in explicit development scripts, never in Cloudflare runtime or normal page rendering. Routes are locale-prefixed and statically generated, with canonical/alternate metadata for SEO.

**Tech Stack:** Astro 5, astro-typst, Typst `.typ` content, Cloudflare adapter, Node/Bun scripts, OpenAI API via `OPENAI_API_KEY`.

---

## Current repository observations

- Articles are loaded from `content/article/**/*.typ` as one `blog` collection in `src/content.config.ts`.
- Article IDs currently equal the path under `content/article/`, e.g. `tutorials/2026/uos-reset-password-and-keyring`.
- Routes are currently unlocalized:
  - `/article/<slug>/`
  - `/article/<slug>.pdf`
  - `/archive/`
  - `/archive/<year>.pdf`
  - `/tag/`
  - `/rss.xml`
- `templates/blog.typ` currently hard-codes `lang: "en"`, while many posts are Chinese. i18n should make `lang` explicit per content file.
- Tags currently have a single source in `templates/enums.typ`; any i18n tag label design must preserve a single canonical tag registry.
- PDF generation is build-time/static and should remain build-time/static.

## Recommended content structure

Use language as the first content path segment and keep the rest of the slug identical across languages:

```text
content/
  article/
    zh/tutorials/2026/uos-reset-password-and-keyring.typ
    en/tutorials/2026/uos-reset-password-and-keyring.typ
    zh/meta/2026/blog-update.typ
    en/meta/2026/blog-update.typ
  page/
    zh/about.typ
    en/about.typ
```

Rationale:

- The same logical article has the same slug in both languages.
- Language switching is path-stable: `/zh/article/x/` <-> `/en/article/x/`.
- Astro content entries remain normal `.typ` files; there is no runtime content lookup or database.
- Translators and reviewers can diff full Typst files naturally.

## Article metadata contract

Extend `shared-template` and `src/content.config.ts` with required i18n fields:

```typst
#show: main.with(
  title: "UOS 重置密码与登录钥匙环修复记录",
  desc: [通过 GRUB 临时 root shell 重置 UOS 用户密码，并处理 deepin 登录钥匙环无法用新密码解锁的问题。],
  date: "2026-05-23",
  tags: (
    blog-tags.linux,
    blog-tags.dev-ops,
    blog-tags.software,
  ),
  lang: "zh",
  i18nKey: "tutorials/2026/uos-reset-password-and-keyring",
  sourceLang: "zh",
  translationStatus: "source",
)
```

Generated counterpart example:

```typst
// @generated by scripts/i18n/translate.mjs
// source: content/article/zh/tutorials/2026/uos-reset-password-and-keyring.typ
// source_sha256: sha256:...

#show: main.with(
  title: "Resetting a UOS Password and Fixing the Login Keyring",
  desc: [Reset a UOS user password from a temporary GRUB root shell and fix the deepin login keyring that can no longer unlock with the new password.],
  date: "2026-05-23",
  tags: (
    blog-tags.linux,
    blog-tags.dev-ops,
    blog-tags.software,
  ),
  lang: "en",
  i18nKey: "tutorials/2026/uos-reset-password-and-keyring",
  sourceLang: "zh",
  translationSourceHash: "sha256:...",
  translationStatus: "machine",
)
```

Rules:

- `i18nKey` is language-neutral and unique per logical article.
- `lang` is the language of the file itself: `zh` or `en`.
- `sourceLang` is the language originally authored by the human.
- `translationStatus` values:
  - `source`: human-authored source file.
  - `machine`: generated by OpenAI and safe to refresh when source changes.
  - `reviewed`: human-reviewed translation; do not overwrite automatically.
- `translationSourceHash` stores the source hash used to generate a machine translation.

## Routing design

Prefer symmetric locale-prefixed URLs:

```text
/zh/
/en/
/zh/article/<slug>/
/en/article/<slug>/
/zh/article/<slug>.pdf
/en/article/<slug>.pdf
/zh/archive/
/en/archive/
/zh/archive/<year>.pdf
/en/archive/<year>.pdf
/zh/tag/<tag>/
/en/tag/<tag>/
/zh/rss.xml
/en/rss.xml
```

No legacy URL compatibility is required for this migration. Remove the old unprefixed route files instead of adding redirects, so the public surface is clean and locale-prefixed only.

Use dynamic route folders rather than duplicating page files:

```text
src/pages/[locale]/index.astro
src/pages/[locale]/about.astro
src/pages/[locale]/article/[...page].astro
src/pages/[locale]/article/[...slug].astro
src/pages/[locale]/article/[...slug].pdf.ts
src/pages/[locale]/archive/index.astro
src/pages/[locale]/archive/[...slug].pdf.ts
src/pages/[locale]/tag/[...tag].astro
src/pages/[locale]/rss.xml.ts
```

Shared helpers should live in:

```text
src/i18n/config.ts       # locales, names, default locale, URL helpers
src/i18n/messages.ts     # UI strings: nav, buttons, page titles, archive labels
src/i18n/content.ts      # localized post filtering, pair lookup, slug parsing
src/i18n/seo.ts          # canonical + hreflang alternate generation
```

## Astro configuration

Keep i18n routing explicit in `src/i18n/config.ts` and the `[locale]` page tree instead of relying on Astro's automatic i18n middleware. This site intentionally removes unprefixed legacy routes, so custom locale-prefixed routes are simpler and avoid hidden redirects.

```ts
// src/i18n/config.ts
export const LOCALES = ["zh", "en"] as const;
export const DEFAULT_LOCALE = "zh";
```

The Astro config should keep `site: "https://dyz.io"` for canonical URL generation, but does not need `i18n.routing` unless the project later wants Astro-managed redirects or fallback routing.

## UI localization

Replace hard-coded UI strings with `messages.ts`:

```ts
export const messages = {
  zh: {
    nav: { home: "首页", posts: "文章", archive: "归档", tags: "标签", about: "关于" },
    article: { downloadPdf: "下载 PDF", lastUpdated: "最后更新于" },
    archive: { title: "归档", description: "博客文章合集" },
  },
  en: {
    nav: { home: "Home", posts: "Posts", archive: "Archive", tags: "Tags", about: "About" },
    article: { downloadPdf: "Download PDF", lastUpdated: "Last updated on" },
    archive: { title: "Archive", description: "Blog article collection" },
  },
} as const;
```

`RootLayout` should receive `locale`, set `<html lang={locale === "zh" ? "zh-CN" : "en"}>`, and pass locale to `Header`, `Footer`, `BaseHead`.

`BaseHead` should emit:

- locale-specific canonical URL;
- `<link rel="alternate" hreflang="zh" ...>`;
- `<link rel="alternate" hreflang="en" ...>`;
- `<link rel="alternate" hreflang="x-default" ...>`;
- `og:locale` and `og:locale:alternate`.

## Translation script design

Create explicit, non-build-time scripts:

```json
{
  "scripts": {
    "i18n:check": "node scripts/i18n/check.mjs",
    "i18n:translate": "node scripts/i18n/translate.mjs",
    "validate": "bun run content:check && bun run tags:check && bun run i18n:check"
  }
}
```

`i18n:translate` should require `OPENAI_API_KEY` and support:

```bash
bun run i18n:translate -- --key tutorials/2026/uos-reset-password-and-keyring --write
bun run i18n:translate -- --from zh --to en --all --write
bun run i18n:translate -- --from en --to zh --all --write
```

Design decisions:

- Default should be dry-run; require `--write` to modify files.
- Translation runs locally/CI-only; OpenAI key is never deployed to Cloudflare runtime.
- Use `OPENAI_TRANSLATION_MODEL` env var with a sensible default; do not hard-code a permanent model policy in content.
- Cache by source hash + model + prompt version.
- Generated files include source path and source hash comments.
- `reviewed` translations are not overwritten unless `--force-reviewed` is explicitly passed.

## Typst-aware translation strategy

Do not send raw Typst to the model and ask it to preserve syntax blindly. Use a segment-aware pipeline:

1. Parse article metadata fields separately:
   - translate `title`;
   - translate `desc` content;
   - copy `date`, `updatedDate`, `tags`, `i18nKey`, `sourceLang`.
2. Protect non-translatable spans with placeholders:
   - fenced code blocks;
   - inline raw/code spans;
   - URLs;
   - image paths;
   - labels/refs;
   - math blocks;
   - Typst command syntax.
3. Translate prose segments, headings, footnote bodies, quote bodies, and link labels.
4. Request strict JSON output:
   ```json
   { "segments": [{ "id": "s1", "translation": "..." }] }
   ```
5. Reconstruct the target Typst file from the original structure.
6. Run `bun run content:check`, `bun run i18n:check`, and `bun run build`.

The translator should be conservative: if a file contains unsupported syntax, it should stop and report the file rather than corrupting it.

## i18n checker design

`scripts/i18n/check.mjs` should fail on:

- a logical `i18nKey` missing either `zh` or `en`;
- duplicate `(lang, i18nKey)` pairs;
- path-derived key mismatch;
- `lang` not matching the first path segment;
- date/tag mismatch across translations;
- machine translation stale because `translationSourceHash` does not match current source hash;
- unresolved `sourceLang` ambiguity;
- UI message key missing in either language;
- localized tag labels missing if the tag registry becomes bilingual.

This checker should be non-mutating and safe to put in `validate` or later `build:strict`.

## Tag registry policy

Preserve one tag source of truth. For bilingual tag labels, the cleanest long-term option is to migrate the canonical tag registry to one structured file and generate Typst enums from it explicitly:

```text
content/tags.json          # canonical: id + zh/en labels
scripts/tags/generate.mjs  # explicit generator for templates/enums.typ
```

Example canonical shape:

```json
{
  "linux": { "zh": "Linux", "en": "Linux" },
  "dev-ops": { "zh": "运维", "en": "DevOps" }
}
```

This should be a separate phase because tags are currently intentionally centralized in `templates/enums.typ`. Do not create a second independent tag table.

## PDF and archive implications

- Article PDFs become locale-scoped: `/zh/article/<slug>.pdf`, `/en/article/<slug>.pdf`.
- Archive PDFs become locale-scoped: `/zh/archive/<year>.pdf`, `/en/archive/<year>.pdf`.
- `buildArchiveSummaries(posts, locale)` should filter by locale and use localized archive titles/descriptions.
- PDF template typography can keep current semantic roles; Noto CJK SC covers Chinese and Latin well enough for bilingual output.

## Comments policy

Decide explicitly whether comments are shared or language-specific.

Recommended default: share comments across translations using the language-neutral `i18nKey` as the comment thread key.

Rationale:

- The logical article is the same content.
- Existing comments can map from old slug to `i18nKey`.
- Splitting comments by locale can be added later by introducing a `locale` column or a compound key.

## Phased implementation

### Phase 0: architecture and safety

- Add `src/i18n/config.ts`, `messages.ts`, `content.ts`, `seo.ts`.
- Extend content schema with optional i18n fields first.
- Add `i18n:check` in warning mode to inventory current articles.
- Do not change URLs yet.

### Phase 1: content model migration

- Move current articles to `content/article/zh/...`.
- Set `lang: "zh"`, `i18nKey`, `sourceLang: "zh"`, `translationStatus: "source"`.
- Fix `templates/blog.typ` so it no longer hard-codes `lang: "en"`.
- Make `i18n:check` fail for missing required metadata.

### Phase 2: translation automation

- Implement `scripts/i18n/translate.mjs` using OpenAI API.
- Generate English counterparts for all Chinese source posts.
- Commit generated translations for reviewable, deterministic builds.
- Add stale-hash detection.

### Phase 3: localized routes

- Add `[locale]` route tree.
- Update layouts/components to accept locale.
- Add language switcher to article/header.
- Add canonical/hreflang metadata.
- Remove the old unprefixed routes instead of adding legacy redirects.

### Phase 4: localized derived artifacts

- Localize archive pages and PDFs.
- Localize article PDFs.
- Localize tag pages and RSS feeds.
- Decide and apply comment thread key policy.

### Phase 5: strict validation

- Add `validate` script: `content:check && tags:check && i18n:check`.
- Optionally add `build:strict` without changing normal `build` until policy is stable.

## Why this architecture is preferable

- Deterministic deploys: Cloudflare serves committed/generated static content, not runtime translations.
- No secret leakage: `OPENAI_API_KEY` is only needed for explicit local/CI translation scripts.
- SEO-correct: both languages have crawlable URLs, canonical links, and hreflang alternates.
- Reviewable: translations are real files in git, so diffs and manual edits are possible.
- Cost-controlled: source hashes prevent retranslating unchanged files.
- Safe for Typst: segment-aware translation avoids corrupting code blocks, links, math, and commands.
- Source-of-truth discipline: article pairs are enforced by `i18nKey`; tags remain single-source; generated files are marked and checked.
- Scalable migration: existing 102 posts can be migrated and translated in batches without requiring a runtime database or CMS.
